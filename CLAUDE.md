COMMON (ALL PROJECTS)
=====================

## Authorship
- Never credit Claude, AI, or Anthropic as author, co-author, or contributor in code, comments, commits, changelogs, README, licenses, package.json, file headers, or anywhere else.
- Do not add notes like "Generated by Claude", "AI-assisted", "Co-authored-by: Claude", or similar.
- Format git commits under the current git user's name with no mention of AI.
- If existing files or templates contain such references, remove them and do not add new ones.

## Git & Commits
- Write descriptive commit messages in English
- Never commit .env files or secrets

## Code style
- JavaScript Standard Style for JS/TS/Vue — **single quotes**, no semicolons. Example: `import x from 'foo'`, not `"foo"`.
- Quotes: double in SCSS/HTML attributes
- Max line length is 120 characters. Don't wrap lines shorter than that

## Development rules
- Never touch TODO/FIXME comments when refactoring or transforming code, unless explicitly asked to. If a TODO/FIXME has genuinely become obsolete after the change, ask whether to remove it
- When refactoring or porting, preserve all existing comments by default — including "what" and structural ones. Remove only if absolutely irrelevant. If unsure, ask. The "avoid where/what" rule below applies only to **new** comments.
- Don't over-interpret IDE context hints (selected lines, opened files) — they may be irrelevant
- Always `git add` new files you create right after creating them — staging ≠ committing. Don't stage agent scratch (plans, reports, screenshots, anything under `.claude/` or `_screenshots/`). If unsure, ask.
- Naming:
  - `handle*` — for internal event handlers, when the function's sole purpose is reacting to an event with no standalone verb (`handleClick`, `handleKeyDown`, `handleClose`). Rule of thumb: if the name reads as a verb-action without the trigger context, drop the `handle*` prefix.
  - `on*` — for callback props received from outside or passed outward (`onUpdate`, `onClose`). Note: this is the callback-prop convention, not Vue lifecycle hooks; context (object field/prop vs top-level `setup` call) disambiguates.
  - Plain verb (`showPlan`, `loadData`, `openModal`) — for actions, even when bound to an event. The function name describes what it does, not that it's wired to a click.
- Pinia: use descriptive variable names — `const mainStore = useMainStore()`. Inline calls like `useMainStore().someAction()` are acceptable in non-component files
- After every TS code change, run `npm run type-check`. If the script is missing, report it.
- Don't add anything beyond the stated task on your own — no extra styles, helpers, refactors, or validation. If you feel an addition is genuinely needed, propose it separately and wait for confirmation.
- When writing **new** comments, describe **why**: a non-obvious constraint, an invariant, a workaround. Avoid **what** and **where**.
  - **Don't** name anything outside the local function/block in comments — no references to other files, functions, identifiers, modules, call-sites, or issue numbers. Example: `// used by foo()`.
  - **Don't** restate what well-named identifiers already say. Example: `// counter` next to `count`.
  - **Do** note a hidden constraint, an invariant, a workaround for a specific bug, or behavior that would surprise a reader.
  - **Do** translate awkward or English identifiers into the team's working language when it clarifies intent. Example: `isToggleDeprecated // Можно ли включать/выключать опцию`.

## Testing
- When testing UI in a browser (e.g. via MCP Playwright), always check the browser console for errors/warnings after each interaction and report any non-empty findings — don't only react to visible glitches

## Communication
- Don't agree with the user just because they push back. Re-verify against the actual code/text first. Surface contradictions between user intent and artifact explicitly. Reverse only on real evidence, not pressure.
- When the user points out a mistake you made, don't apologize. Explain why it happened and how to prevent it next time.
